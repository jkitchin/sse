I still suggest that in the beginning you focus on setuptools until you have a need that it does not meet. The packages we have made so far are compatible with setuptools. See the quickstart to get started.

Here I note some of the more common things you might want to do with packages and setuptools.

Package configuration¬∂

We focused on setup.py as the primary configuration file, but there are several other ways you could do it. The field is moving towards pyproject.toml as a standard way to specifying project metadata.

setuptools also supports an older setup.cfg file.

Technically, setup.py is being moved away from, mostly for security reasons; setup.py allows arbitrary code to be executed during build. The files above are declarative and don‚Äôt run code.

Package discovery¬∂

We have usually specified the modules your package provides explicitly. You can rely on ‚Äúdiscovery‚Äù though to automatically generate this list. See package discovery.

Dependency management¬∂

Your package may depend on other libraries, even specific versions of those libraries. You can specify these dependencies in the setup files to ensure they get installed with your package. See dependency management.

Entry points - aka shell commands¬∂

If your package provides shell commands, you can specify these as ‚Äúentry points‚Äù. See entry points.

Data files¬∂

If your package relies on data files, e.g. a csv file of data that is used, or templates of some kind, etc. then you need to include them in your package. See datafiles for how to do that.

Distributing your package¬∂

For others to use your package they need a way to get it. You can zip the source directory up and email it to them, put it on some public website for download, etc.

Alternatively, you can use methods compatible with pip to allow people to install it.

GitHUB¬∂

You can always distribute your package via GitHUB (this is one form of a public website). You can even install packages straight from GitHUB. See pycse for an example. Here is a command to install the current version of pycse.

pip install git+git://github.com/jkitchin/pycse

You can install specific versions from tags:

pip install https://github.com/jkitchin/pycse/archive/refs/tags/v1.0.zip

See VCS support for details on other variations, e.g. a branch name, or a commit hash.

The upside of this is you don‚Äôt have to do anything special beyond getting it to GitHUB! The downside is it is not as simple as pip install pycse, and some people might not want to install straight from gitHUB for ‚Äúsecurity reasons‚Äù. This is an illusion though, the version of pycse on pypi is the same as the one in GitHUB‚Ä¶

pypi¬∂

You can also publish your package to PyPi. Then anyone can pip install your package by its name.

twine is a Python package to help you with this process. Note that you need to have an account with PyPi and get an API token from them. Then follow the directions here.

End of the beginning¬∂

For more than 20 years Python packaging has evolved, and you can count on it continuing to evolve in the future. For the majority of needs, I think setuptools will serve you well. The other options were made to fulfill specific needs, and if you don‚Äôt have them, you don‚Äôt need them. It is worth learning enough about these other options to know if they solve a particular problem for you. You may also run into projects that use them, and have to adapt your workflow to use them.
